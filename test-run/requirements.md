# AI 测试用例生成系统 需求文档

> 版本：v1.0 | 更新日期：2026-07-14

## 1. 系统概述

AI 测试用例生成系统是一套基于 AI 技术的自动化测试用例生成平台，通过 7 个可串联的 Skill 环节，实现从需求文档到测试报告的全流程自动化。系统支持知识库 RAG 增强，能够基于业务规则、历史用例和线上坑点生成高质量的测试用例。

## 2. 用户管理模块

### 2.1 用户注册

**功能描述：** 新用户通过手机号注册系统账号。

**业务规则：**
- 注册方式：手机号 + 短信验证码
- 手机号格式：中国大陆 11 位手机号，以 1 开头，第二位为 3-9
- 验证码：6 位数字，有效期为 5 分钟
- 验证码发送间隔：同一手机号 60 秒内不可重复发送
- 密码要求：8-20 位，必须包含字母和数字，区分大小写
- 同一手机号 24 小时内最多发送验证码 5 次
- 同一手机号最多注册 1 个账号

**注册流程：**
1. 用户输入手机号
2. 系统校验手机号格式
3. 用户点击"获取验证码"
4. 系统发送短信验证码（有效期 5 分钟）
5. 用户输入验证码和密码
6. 系统验证通过后完成注册

### 2.2 用户登录

**功能描述：** 已注册用户通过多种方式登录系统。

**登录方式：**
- 账号密码登录
- 手机验证码登录
- 第三方登录（微信、支付宝）

**业务规则：**
- 密码连续输入错误 5 次，账号锁定 30 分钟
- 账号锁定期间，验证码登录不受影响
- Token 有效期为 2 小时，支持自动续期
- 同一账号最多在 3 台设备同时登录
- 第 4 台设备登录时，最早登录的设备自动下线

### 2.3 密码找回

**功能描述：** 用户忘记密码时通过手机号重置密码。

**业务规则：**
- 重置密码需要验证手机验证码
- 新密码不能与最近 3 次密码相同
- 重置密码后，所有设备的 Token 失效
- 重置密码链接/验证码有效期为 10 分钟

## 3. 项目管理模块

### 3.1 创建项目

**功能描述：** 用户创建测试项目，管理测试用例。

**业务规则：**
- 项目名称：1-50 个字符，不允许特殊字符
- 项目描述：最多 500 个字符
- 每个用户最多创建 20 个项目
- 项目名称在同一用户下不可重复

**创建流程：**
1. 用户填写项目名称和描述
2. 系统校验项目名称唯一性
3. 创建项目，生成唯一项目 ID

### 3.2 项目配置

**功能描述：** 配置项目的测试维度和知识库关联。

**可配置项：**
- 测试维度：正向测试、负向测试、边界测试、异常测试（默认全选）
- 增强维度：性能测试、安全测试（默认不选）
- 知识库路径：关联 Obsidian Vault 路径
- 输出格式：Excel / XMind / 两者
- 优先级体系：P0/P1/P2

## 4. 需求分析模块

### 4.1 需求文档上传

**功能描述：** 用户上传 Markdown 格式的需求文档。

**业务规则：**
- 支持 .md / .markdown 格式
- 文件大小限制：单个文件最大 5MB
- 支持 UTF-8 / GBK / GB2312 编码
- 自动检测并提示编码问题

### 4.2 需求自动分析

**功能描述：** AI 自动拆解需求文档，识别功能模块、功能点和待确认事项。

**输出内容：**
- 功能模块清单
- 每个模块的功能点
- 每个功能点的可测项
- 待确认事项（高/中/低优先级）

**分析规则：**
- 模糊描述识别：检测"一定的""若干""大概"等不具体词汇
- 信息缺失检测：数值、时间、金额等缺失具体数字
- 逻辑矛盾检测：前后描述不一致的地方
- 异常场景缺失：正常流程完整但缺少异常处理

## 5. 测试用例生成模块

### 5.1 测试点生成

**功能描述：** 基于需求分析结果，自动生成结构化测试点清单。

**生成规则：**
- 每个功能点至少 4 个测试点（正向1、负向1、边界1、异常1）
- 总测试点数量控制在功能点数量的 4-8 倍
- 正向测试占比不超过 30%
- 各维度至少各 1 个测试点

**测试维度：**
| 维度 | 说明 | 是否默认 |
|------|------|---------|
| 正向测试 | 正常业务流程、合法输入 | 是 |
| 负向测试 | 非法输入、异常操作 | 是 |
| 边界测试 | 数值/长度/数量边界 | 是 |
| 异常测试 | 网络/服务/并发异常 | 是 |
| 性能测试 | 响应时间、并发、大数据量 | 否 |
| 安全测试 | 越权、注入、篡改、泄露 | 否 |

### 5.2 用例文件生成

**功能描述：** 将测试点转化为可执行的测试用例文件。

**输出格式：**
- Excel（.xlsx）：12 列结构化表格
- XMind（.xmind）：树状脑图

**Excel 列结构：**
| 列 | 字段 | 说明 |
|----|------|------|
| A | 用例编号 | TC-001 格式 |
| B | 所属模块 | 功能模块名称 |
| C | 功能点 | 功能点名称 |
| D | 测试维度 | 正向/负向/边界/异常/性能/安全 |
| E | 用例标题 | 简洁描述 |
| F | 优先级 | P0/P1/P2 |
| G | 前置条件 | 执行前必要条件 |
| H | 测试步骤 | 编号步骤 |
| I | 测试数据 | 具体测试数据 |
| J | 预期结果 | 具体预期结果 |
| K | 备注 | 额外说明 |
| L | 执行结果 | 人工填写 |

**优先级分配规则：**
- P0：核心业务流程正向测试、涉及资金/安全的测试
- P1：一般功能正向测试、大部分负向测试、关键边界测试
- P2：非核心功能测试、UI 细节测试

## 6. 用例评审模块

### 6.1 自动评审

**功能描述：** 对生成的测试用例进行四维质量检查。

**四个维度：**
1. **缺失用例检测**：核心场景、边界条件、异常路径、权限场景
2. **质量问题识别**：描述含糊、步骤不清晰、预期结果模糊
3. **重复冗余检查**：用例相似度 > 80% 视为重复
4. **整改建议生成**：补充、优化、合并建议

**质量评分标准：**
| 维度 | 满分 |
|------|------|
| 完整性 | 30 |
| 清晰性 | 30 |
| 准确性 | 20 |
| 可执行性 | 20 |

## 7. 知识库管理模块

### 7.1 知识库检索（RAG 增强）

**功能描述：** 通过 MCP 协议访问 Obsidian Vault，检索业务规则和历史坑点。

**知识分类：**
| 分类 | 目录 | 说明 |
|------|------|------|
| 历史优质用例 | 历史用例 | 评审通过的优质用例 |
| 业务规则 | 业务规则 | 字段限制、校验规则 |
| 线上问题沉淀 | 线上坑点 | 历史 Bug、易漏场景 |
| 用例模板 | 用例模板 | 编写模板、命名规范 |
| 数据字典 | 数据字典 | 表字段、枚举值 |
| 业务规范 | 业务规范 | 功能说明、状态流转 |
| 团队规范 | 团队规范 | 断言规则、测试流程 |

**检索规则：**
- 使用 BM25 文本匹配算法排序
- 支持按分类过滤检索
- 返回条数可配置（默认 20 条）

### 7.2 知识回灌

**功能描述：** 将评审通过的优质产物沉淀到知识库。

**回灌内容：**
- 优质用例（Excel 格式）
- 确认的业务规则（从需求分析提取）
- 失败用例分析（作为坑点）

**命名规则：** `日期前缀_标题.md`（如 `20260714_支付回调重复.md`）

## 8. 测试报告模块

### 8.1 报告自动生成

**功能描述：** 根据测试执行结果，生成结构化测试质量报告。

**报告内容：**
1. 总体概览（总数、通过率、评级）
2. 模块通过率分布
3. 优先级分析（P0/P1/P2 通过率）
4. 测试维度分析
5. 失败用例分析（原因分类）
6. 阻塞用例分析
7. 风险评估（高/中/低）
8. 测试结论与发布建议

**质量评级标准：**
| 通过率 | 评级 | 说明 |
|--------|------|------|
| ≥ 95% | 优秀 | 可发布 |
| 85%-95% | 良好 | 修复少量问题后可发布 |
| 70%-85% | 中等 | 需修复较多问题 |
| < 70% | 较差 | 不建议发布 |

## 9. Pipeline 全流程模块

### 9.1 全流程串联

**功能描述：** 将 7 个环节自动串联为一键 pipeline。

**执行模式：**
| 模式 | 说明 |
|------|------|
| 全自动 | 7 步连续执行不暂停 |
| 半自动 | 关键节点暂停确认（推荐） |
| 单步执行 | 每步都暂停确认 |

**人工检查点（半自动模式）：**
1. Step 2 后 — 确认需求分析结果
2. Step 3 后 — 确认测试点覆盖
3. Step 4 后 — 确认用例质量
4. Step 5 后 — 确认评审建议
5. Step 6 — 必须人工执行测试

### 9.2 断点续跑

**功能描述：** 支持从中断处恢复执行。

**规则：**
- 检查输出文件是否存在，自动跳过已完成步骤
- 支持查看当前进度
- 支持从断点继续执行

## 10. 待确认事项

### 10.1 需要产品确认的内容

1. **多语言支持**：系统是否需要支持英文界面？需求文档是否支持英文输入？
2. **团队协作**：是否支持多人同时编辑同一项目的用例？冲突如何处理？
3. **版本管理**：需求文档版本变更后，已生成的用例如何处理？是否需要自动对比变更？
4. **API 接口**：是否需要提供 API 接口供外部系统调用？接口鉴权方式？
5. **数据导出**：除 Excel/XMind 外，是否需要支持导出为 TestRail / Zephyr 等测试管理工具格式？
