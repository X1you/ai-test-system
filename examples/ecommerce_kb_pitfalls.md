# 线上坑点沉淀：电商订单系统历史故障

> 本文档记录电商订单系统历史上发生过的线上 Bug 和易漏场景，供测试设计参考。

## KB-库存超卖：高并发下单导致库存为负

**故障时间**：2025-03-15
**影响**：大促期间 100 件库存的商品被卖出 127 件，导致超卖
**根因**：
- 库存校验和库存扣减不是原子操作（先 SELECT 再 UPDATE）
- 高并发下多个请求同时读到库存=1，都校验通过，都执行扣减
**修复**：改为 `UPDATE stock SET num = num - 1 WHERE id = ? AND num >= 1`（乐观锁/条件更新）
**测试要点**：
- 必须验证并发场景下的库存一致性
- 模拟 100 并发抢 1 件库存的最后 1 个
- 验证扣减后库存绝不会为负数

## KB-浮点数对账不平：金额计算精度丢失

**故障时间**：2025-04-22
**影响**：每日财务对账时，订单总额与支付流水总和存在 0.01~0.03 元误差，累计一个月差几百元
**根因**：
- 使用 float 类型存储金额（如 19.99 元在二进制浮点表示为 19.989999999...）
- 多次乘法/加法运算后精度累积误差
- 优惠券折扣计算：19.99 × 0.8 = 15.992，round 后 15.99，但不同位置 round 时机不同导致不一致
**修复**：
- 金额统一用整数分存储（19.99 元 → 1999 分）
- 所有运算用 Decimal 类型
- 折扣计算只在最终结算时 round 一次
**测试要点**：
- 必须验证含小数的金额计算（如 19.99 × 数量 3、折扣 8 折）
- 验证订单总额 = Σ商品金额 + 运费 - 优惠，精确到分
- 验证连续多笔订单的累加和对账

## KB-支付回调重复处理：订单被支付两次

**故障时间**：2025-05-08
**影响**：微信支付因网络问题重试回调 3 次，系统未做幂等处理，导致同一订单状态被更新多次，库存被重复扣减
**根因**：
- 支付回调接口未做幂等校验
- 收到回调直接 UPDATE 订单状态，未检查当前状态
**修复**：
- 回调处理前先检查订单当前状态，只有"待支付"才能改为"已支付"
- 用订单号+支付流水号做唯一约束
**测试要点**：
- 必须验证重复回调的幂等性（同一回调发 3 次，订单状态不变）
- 验证回调乱序到达（先收到失败回调后收到成功回调）的处理
