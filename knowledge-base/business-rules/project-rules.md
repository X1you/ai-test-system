# 业务规则

## AI 测试用例生成系统 — 业务规则

### 1. 需求分析规则
**模块：** 需求分析
- 需求文档必须为 Markdown 格式
- 必须识别至少 3 个模块、10 个功能点、30+ 可测项
- 待确认事项必须标注优先级（高/中/低）
- 发现逻辑矛盾必须向用户指出

### 2. 测试点梳理规则
**模块：** 测试点梳理
- 测试点结构：模块 → 功能点 → 测试维度 → 具体测试点
- 每个测试点必须包含描述、测试数据、预期结果
- 维度平衡：正向 ≤ 30%，负向/边界/异常各 ≥ 20%
- 优先级建议：核心流程正向 → P0，其余正向 → P1
- 安全测试触发条件：涉及权限、资金、敏感数据
- 性能测试触发条件：涉及并发、大量、响应时间

### 3. 测试用例生成规则
**模块：** 测试用例生成
- Excel 必须包含 12 列完整结构
- XMind 必须使用纯标准库实现（zipfile + json）
- Excel 格式：表头深蓝白字、P0 红/P1 橙/P2 绿
- 步骤描述必须具体，避免"按照业务流程执行"的笼统表述
- 已有执行结果的 Excel 不能被覆盖

### 4. 用例评审规则
**模块：** 用例评审
- 四维质检：缺失用例/质量问题/重复冗余/整改建议
- 质量评分：完整性 30 分 + 清晰性 30 分 + 准确性 20 分 + 可执行性 20 分
- 整改清单必须包含优先级

### 5. 测试报告规则
**模块：** 测试报告生成
- 必须包含：总体概览、模块通过率、优先级分析、失败分析、风险评估
- 通过率 = 通过 ÷ (通过 + 失败 + 阻塞 + 跳过)
- 发布建议算法：P0 失败 → 不建议发布，通过率 ≥ 95% → 建议发布
- 质量评级：≥95% 优秀、≥85% 良好、≥70% 中等、<70% 较差

### 6. 全流程串联规则
**模块：** 全流程串联
- 7 步按顺序执行，不可跳过步骤
- Step 6（人工执行测试）不可自动跳过
- 断点续跑：已完成的步骤自动跳过
- 知识库在 Step 2 检索、Step 5+7 回灌
