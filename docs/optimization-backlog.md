## 2. 性能优化

### P-01: LLM 调用缓存层 [已完成 2026-07-16]

| 字段 | 内容 |
|------|------|
| **具体内容** | 当前 LLM 调用（`core/llm_client.py`）无缓存机制。在以下场景中，相同的 Prompt 会被多次调用：<br>1. 断点续跑时重复执行已完成的步骤<br>2. 同一份需求文档多次运行 Pipeline<br>3. 知识库检索的重复查询<br>LLM 调用成本高（耗时 2-10s/次，Token 计费），缓存可显著优化。 |
| **优先级** | P1（重要） |
| **预期效果** | 重复 LLM 调用减少 50%+，Pipeline 执行时间缩短 30-50%，Token 消耗降低。 |
| **所需资源** | 3-5 人·天 |
| **潜在风险** | 中 — 缓存 Key 设计不当可能导致返回过期结果；缓存未命中时性能无改善。需注意 Prompt 的确定性（temperature=0 时可安全缓存）。 |
| **实施路径** | 1. 在 `core/llm_client.py` 中添加缓存层（`LLMCache` 类）<br>2. 缓存 Key = `hash(model + system + prompt + temperature)`<br>3. 使用 SQLite 或内存 LRU 缓存（`cachetools`）<br>4. 缓存 TTL 默认 24 小时，可配置<br>5. 在 `LLMGateway` 中集成缓存查询（先查缓存 → 缓存未命中 → 调用 LLM → 写入缓存）<br>6. 添加 `--no-cache` CLI 参数以绕过缓存 |

### P-02: 数据库索引优化 [已完成 2026-07-16]

| 字段 | 内容 |
|------|------|
| **具体内容** | 当前数据模型（`db/models.py`）中仅 `User.username` 和 `User.api_key` 有索引。缺少以下关键索引：<br>1. `pipelines.status` — 列表页按状态过滤<br>2. `pipeline_steps.pipeline_id` — 按 Pipeline 查询步骤<br>3. `pipeline_steps.status` — 按状态筛选步骤<br>4. `artifacts.pipeline_id` — 按 Pipeline 查询产物<br>随着数据量增长（>1000 条记录），全表扫描性能将显著下降。 |
| **优先级** | P1（重要） |
| **预期效果** | 列表查询响应时间从全表扫描 O(n) 降至索引查找 O(log n)，提升 10-100 倍。 |
| **所需资源** | 1 人·天 |
| **潜在风险** | 低 — 写操作有轻微性能开销（<5%），但 SQLite 下可忽略。需注意迁移脚本的兼容性。 |
| **实施路径** | 1. 在 `db/models.py` 中添加 `__table_args__` 定义复合索引<br>2. 创建 Alembic 迁移脚本<br>3. 索引建议：<br>   - `pipelines`: `(status, started_at)` 复合索引<br>   - `pipeline_steps`: `(pipeline_id, step_id)` 唯一复合索引<br>   - `artifacts`: `pipeline_id` 索引<br>4. 使用 `EXPLAIN QUERY PLAN` 验证索引生效 |

### P-03: 前端静态资源构建压缩 [已完成 2026-07-16]

### P-05: 异步 LLM 调用优化 [已完成 2026-07-16]

| 字段 | 内容 |
|------|------|
| **具体内容** | 当前 `LLMClient.chat()` 是同步方法，通过 `asyncio.to_thread` 包装为异步（`llm_gateway.py:93`）。但 `OpenAI` SDK 本身支持异步调用（`AsyncOpenAI`），使用同步包装增加线程切换开销。 |
| **优先级** | P2（建议） |

| **实施路径** | 1. 在 `LLMClient` 中增加 `async_chat()` 方法，使用 `AsyncOpenAI`<br>2. 保留 `chat()` 同步方法供 CLI/线程使用<br>3. `LLMGateway.chat()` 改为直接调用 `async_chat()`<br>4. 确保 `Pipeline` 中的调用不受影响 |
