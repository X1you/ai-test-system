## 🔵 P3-A：Step3 测试点模块覆盖不全

### 问题描述

需求分析识别出 **11 个模块**，但 Step3 测试点梳理只覆盖了 **4 个核心模块**（Pipeline编排/知识库/需求分析/测试点梳理），缺失 7 个模块（用例生成/评审/报告/外部集成/UI/配置/认证）。

### 根因分析

Step3 的 prompt 把整个需求分析文档一次性发给 LLM，当文档较大（本次 46KB / 1107 行 / 11 模块）时：
1. **Token 预算挤压**：LLM 为了控制输出长度，主动"聚焦"前几个模块，后面的模块被压缩或忽略
2. **注意力衰减**：长上下文中靠后的内容权重降低
3. **单次调用限制**：一次 LLM 调用难以同时保证广度（覆盖所有模块）和深度（每模块6维度）

### 方案对比

| 方案 | 做法 | 优点 | 缺点 | 工作量 |
|------|------|------|------|--------|
| **A. Prompt 硬约束** | 在 prompt 加"必须覆盖所有模块"+ 分模块计数自检 | 改动小 | LLM 可能仍忽略，或为满足约束降低每模块深度 | 小（改 prompt） |
| **B. 分模块调用** | Step3 按模块数分片，每模块独立调用 LLM | 彻底解决，每模块都能深度展开 | LLM 调用次数 ×N，耗时和成本上升 | 中（重构 step3） |
| **C. 两阶段生成** | 第一阶段生成模块大纲，第二阶段逐模块填充 | 平衡覆盖与深度 | 架构改动大 | 大（新增阶段） |

### 推荐方案：B（分模块调用）

**理由**：
- 彻底解决覆盖问题，不依赖 LLM 自觉性
- 成本可控：11 模块 × 单次调用 ≈ 11 × 30秒 = 5.5 分钟（vs 现在单次 30 秒但只覆盖 4 模块）
- 每模块独立 prompt 可以塞入更详细的该模块需求分析，深度更好

**实施步骤**：
1. Step3 先解析 `requirements_analysis.md` 提取模块列表（已有结构化格式）
2. 对每个模块独立调用 LLM 生成测试点（并发，受 `pipeline.max_concurrent` 控制）
3. 合并所有模块的测试点为统一 `testpoints.md`，重新编号
4. 自检：校验模块数 == 需求分析模块数

**风险**：
- 并发调用可能触发 LLM 限流（DeepSeek 有 RPM 限制）
- 合并时编号需要全局重排，避免冲突

## 🔵 P3-B：Step7 报告失败原因推断不准

### 问题描述

`generate_report.py` 的失败原因推断基于 11 类关键词匹配（权限/认证/接口/数据/状态/边界...），但对"代码审查型失败"匹配不准：
- TC-006（代码缺 hash 校验）被推断为"权限/认证问题"
- TC-010（功能未实现）被推断为"接口/服务异常"
- TC-016（依赖功能缺失）被推断为"接口/服务异常"

### 根因

11 类分类是为"功能测试失败"设计的，假设失败都有运行时表现（报错/超时/返回码）。但本次 Step6 有大量"代码审查型失败"——通过阅读代码发现功能缺失或逻辑缺陷，这类失败没有运行时表现，关键词匹配失效。

### 方案

**扩充失败原因分类**，新增 2 类：

```
新增「功能未实现」类：
  关键词：未发现|不存在|未实现|未支持|该功能|规划中|未承诺
  修复建议：确认该功能是否在需求范围内；若在则补实现，若不在则修正用例预期

新增「代码缺陷」类：
  关键词：未校验|缺失|无校验|不验证|漏掉|缺少|未检查
  修复建议：补充对应的校验/检查逻辑，参考用例的预期结果
```

**实施位置**：`scripts/generate_report.py` 的失败原因推断函数（搜 `失败原因` 或 `reason`）

**工作量**：约 30 行，改分类映射表 + 关键词正则

### 验证方式

修复后重新生成报告，检查 TC-006/010/016/018 的失败原因是否从"权限/认证/接口"变为"代码缺陷/功能未实现"。

## 📌 后续观察项

以下问题本次未深入，建议后续关注：

1. **YAML frontmatter 格式不规范**：回灌的知识文件 YAML 用 `{key: value}` 花括号格式而非标准 `key: value`，虽然 Obsidian 容忍但不规范。位置：`kb_manager_mcp.py` 的文件写入逻辑。

2. **回灌统计未推送到 WebUI progress**：`ingested_cases_count` / `ingested_pitfalls_count` 存在 state 但未加入 `get_progress()` 返回字段，前端只能从日志看到回灌结果。如需在进度面板展示，改 `web/services/pipeline_task.py:get_progress()`。

3. **LLM 输出稳定性**：相同需求多次跑，测试点数量和模块覆盖有波动（第一次 11 模块/82 测试点，第二次 9 模块/78 测试点）。根因是 temperature=0.3 的随机性。如需稳定输出可降到 0.1，但会降低多样性。
