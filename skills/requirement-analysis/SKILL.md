# 需求分析 Skill

将需求文档（Markdown 格式）拆解为结构化的功能模块清单，自动识别：
- 功能模块
- 每个模块的功能点
- 每个功能点的可测项
- 待确认事项（描述模糊、信息缺失、规则不明确）

## 触发条件

当用户说以下内容时，使用本 Skill：
- "分析这个需求"
- "拆解需求文档"
- "需求文档分析"
- "这个需求有什么问题"
- 带有 "requirements.md"、"需求文档" 等关键词

## 执行步骤

### 1. 读取需求文档
使用 `read_file` 工具读取用户提供的 Markdown 需求文档路径。

### 2. 分析并生成两个文件

#### 文件 1: requirements_analysis.md（需求拆解）

**格式要求：**
```markdown
# 需求拆解分析

*由 AI 自动生成，基于需求文档内容*

## 模块一：[模块名称]
- 功能点 1.1：[功能点名称]
  - 可测项：[可测项1]、[可测项2]、[可测项3]
- 功能点 1.2：[功能点名称]
  - 可测项：[可测项1]、[可测项2]

## 模块二：[模块名称]
...
```

**提取规则：**
- 从文档中识别独立的功能模块（如用户管理、订单管理、支付系统）
- 每个模块下提取具体功能点（如注册、登录、查询、删除）
- 每个功能点列出可测项（输入验证、流程完整性、边界条件）

#### 文件 2: clarification_needed.md（待确认事项）

**格式要求：**
```markdown
# 待确认事项清单

以下事项需要与产品/开发确认：

## [模块名称]

1. **[事项标题]**
   - 问题：[描述不清楚/信息缺失的地方]
   - 建议：[建议的产品/开发需要明确的内容]
   - 优先级：[高/中/低]
   - 影响范围：[影响的模块/功能]

2. **[事项标题]**
   - 问题：[描述不清楚/信息缺失的地方]
   - 建议：[建议的产品/开发需要明确的内容]
   - 优先级：[高/中/低]
   - 影响范围：[影响的模块/功能]
```

**识别规则：**
- 模糊描述："一定的"、"若干"、"等"、"可能"、"大概"等不具体词汇
- 信息缺失：数值、时间、次数、金额等没有具体数字
- 规则不明确：错误处理、限制条件、边界情况没有说明
- 文档中明确标注的"需要确认"、"待定"、"TODO"内容
- 逻辑矛盾：前后描述不一致的地方
- 缺失场景：正常流程描述完整，但缺少异常流程

### 3. 使用 write_file 工具保存文件
将两个文件保存到用户指定的输出目录（默认为当前目录）

### 4. 向用户汇报

**汇报格式：**
```markdown
✅ 需求分析完成！

📊 分析统计：
- 文档行数：N 行
- 功能模块：N 个
- 功能点：N 个
- 可测项：N 个
- 待确认事项：N 个

📁 生成的文件：
- requirements_analysis.md - 需求拆解（N KB）
- clarification_needed.md - 待确认清单（N KB）

⚠️ 待确认事项优先级分布：
- 高优先级：N 个（影响核心流程）
- 中优先级：N 个（影响功能完整性）
- 低优先级：N 个（优化建议）

💡 下一步：
请先与产品/开发确认待确认事项，特别是高优先级项。
确认后可以继续进行测试点梳理。
```

### 5. 生成质量报告（可选）

如果用户要求更详细的分析，可以生成第三个文件：
- `requirements_quality_report.md` - 需求质量评估报告

包含：
- 文档完整性评分
- 模块覆盖率
- 描述清晰度评分
- 可测试性评分
- 改进建议

## 输入参数

在执行时需要确认：
- 需求文档路径（必需）
- 输出目录（可选，默认为当前目录）
- 是否生成质量报告（可选，默认否）

## 高级功能

### 1. 多文档合并分析
支持同时分析多个需求文档，生成统一的分析结果。

### 2. 历史对比
如果用户提供了新版本的需求文档，可以对比历史版本，识别变更点。

### 3. 自动标注优先级
根据以下规则自动标注待确认事项的优先级：
- **高优先级**：影响核心业务流程、涉及资金/安全、阻塞测试设计
- **中优先级**：影响功能完整性、边界情况、错误处理
- **低优先级**：优化建议、UI细节、非核心功能

### 4. 依赖关系分析
识别模块间的依赖关系，在输出中标注前置依赖。

## 注意事项

1. 如果需求文档质量较差，待确认清单可能很长，这是正常的
2. 必须生成两个独立的 Markdown 文件
3. 用户应与产品/开发确认待确认项后再继续测试点梳理
4. 可测项要具体，避免"验证功能"这种模糊表述
5. 对于高优先级的待确认事项，建议在汇报中特别标注
6. 如果发现明显的逻辑矛盾，需要立即向用户指出

## 常见问题

**Q: 待确认事项太多怎么办？**
A: 优先处理高优先级事项，中低优先级可以在测试过程中逐步确认。

**Q: 需求文档格式不规范怎么办？**
A: Skill 会尽力提取，但建议用户先整理文档格式。对于严重不规范的文档，会提示用户。

**Q: 如何确定优先级？**
A: Skill 会自动标注，用户也可以根据实际情况手动调整。

**Q: 分析结果不准确怎么办？**
A: 可以提供更多上下文信息，或者明确告诉 AI 需要关注哪些方面。
