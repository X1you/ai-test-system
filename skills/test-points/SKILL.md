# 测试点梳理 Skill

基于已分析的需求文档，生成详细的测试点清单，为后续测试用例生成做准备。

## 触发条件

当用户说以下内容时，使用本 Skill：
- "梳理测试点"
- "生成测试点清单"
- "测试点分析"
- "性能测试点"
- "安全测试点"
- 带有 "testpoints"、"测试点" 等关键词

## 执行步骤

### 0. 知识库检索增强（可选但推荐）

在生成测试点前，先检索知识库获取业务规则和历史坑点：

```bash
HERMES_PYTHON="$HOME/.hermes/hermes-agent/venv/bin/python"
KB="$HOME/.hermes/skills/testing/knowledge-base/scripts/kb_manager.py"
$HERMES_PYTHON "$KB" export "需求关键词" --output knowledge-context.md
```

将导出的 `knowledge-context.md` 作为额外参考，特别关注：
- 📋 业务规则：确保测试点覆盖所有字段限制和校验规则
- ⚠️ 线上坑点：将历史 Bug 对应的场景纳入测试点

> 如果知识库未初始化，跳过此步骤即可。

### 1. 读取已分析的需求文档
使用 `read_file` 工具读取 `requirements_analysis.md` 文件。

### 2. 确认测试维度配置

默认包含的测试维度：
- 正向测试（必需）
- 负向测试（必需）
- 边界测试（必需）
- 异常测试（必需）

可选测试维度：
- 性能测试（可选，默认关闭）
- 安全测试（可选，默认关闭）

**配置方式：**
```
# 包含性能测试
梳理测试点，包含性能测试

# 包含安全测试
梳理测试点，包含安全测试

# 包含所有测试维度
梳理测试点，包含性能和安全测试

# 只生成性能和安全测试
梳理测试点，只生成性能和安全测试
```

### 3. 生成测试点清单

**输出文件：testpoints.md**

**格式要求：**
```markdown
# 测试点清单

*由 AI 自动生成，基于需求分析文档*

## 模块一：[模块名称]

### 功能点 1.1：[功能点名称]

#### 测试维度：正向测试
- 测试点 1.1.1：[具体测试点描述]
  - 测试数据：[测试数据示例]
  - 预期结果：[预期结果]

- 测试点 1.1.2：[具体测试点描述]
  - 测试数据：[测试数据示例]
  - 预期结果：[预期结果]

#### 测试维度：负向测试
...

#### 测试维度：边界测试
...

#### 测试维度：异常测试
...

#### 测试维度：性能测试（如果包含）
- 测试点 1.1.N：响应时间测试
  - 测试数据：正常操作
  - 预期结果：响应时间 < 500ms

- 测试点 1.1.N+1：并发测试（100个并发）
  - 测试数据：100个用户同时操作
  - 预期结果：成功率 > 99%

#### 测试维度：安全测试（如果包含）
- 测试点 1.1.N+2：越权访问测试
  - 测试数据：用户A访问用户B的数据
  - 预期结果：提示"无权访问"

- 测试点 1.1.N+3：数据篡改测试
  - 测试数据：修改关键数据
  - 预期结果：数据验证失败
```

### 4. 测试维度说明

#### 基础测试维度（4个，必需）

**正向测试**
- 正常业务流程
- 正常数据输入
- 预期成功场景

**负向测试**
- 错误数据输入
- 非法操作
- 异常流程

**边界测试**
- 数据边界（最小值、最大值）
- 数量边界（0、1、最大数量）
- 时间边界（最早、最晚）

**异常测试**
- 网络异常
- 服务异常
- 并发冲突
- 数据异常

#### 增强测试维度（2个，可选）

**性能测试**

覆盖关键功能：
- 响应时间测试
- 并发压力测试
- 大数据量测试
- 稳定性测试

关键指标：
- 响应时间：< 500ms（关键操作）、< 1s（一般操作）
- 并发数：100（普通）、1000（查询类）、10000（导出类）
- 成功率：> 99%（关键操作）、> 95%（一般操作）
- 数据量：100条（小）、1000条（中）、10000条（大）、100万条（超大）

**安全测试**

覆盖高风险场景：
- 越权访问测试
- 数据篡改测试
- 注入攻击测试（SQL注入、XSS、CSRF）
- 重复攻击测试
- 信息泄露测试

关键测试：
- 权限控制：用户只能访问自己的数据
- 数据验证：防止非法数据提交
- 签名验证：支付回调、接口签名
- 频率限制：防止恶意操作
- 脱敏处理：敏感信息不泄露

## 测试点生成规则

### 1. 正向测试规则
- 覆盖所有正常业务流程
- 覆盖所有合法输入范围
- 覆盖所有成功场景

### 2. 负向测试规则
- 覆盖所有非法输入
- 覆盖所有错误操作
- 覆盖所有异常流程

### 3. 边界测试规则
- 数值边界（0、1、-1、最大值、最小值）
- 长度边界（空、1、最大长度、最大长度+1）
- 时间边界（边界时间点）
- 数量边界（0、1、最大数量、最大数量+1）

### 4. 异常测试规则
- 网络异常（超时、断网）
- 服务异常（服务宕机、响应慢）
- 并发冲突（同时操作同一数据）
- 数据异常（数据丢失、数据错误）

### 5. 性能测试规则（可选）

#### 响应时间测试
- 关键操作：< 500ms
- 一般操作：< 1s
- 复杂操作：< 3s
- 数据导出：< 10s

#### 并发测试
- 写操作（下单、支付）：100并发
- 读操作（查询）：1000并发
- 导出操作：10并发

#### 大数据量测试
- 小数据量：100条
- 中数据量：1000条
- 大数据量：10000条
- 超大数据量：100万条

#### 稳定性测试
- 持续操作：重复1000次
- 长时间运行：持续1小时

### 6. 安全测试规则（可选）

#### 越权访问测试
- 用户A访问用户B的数据
- 普通用户执行管理员操作
- 未登录用户访问需要认证的数据

#### 数据篡改测试
- 修改订单金额
- 修改库存数据
- 修改支付金额

#### 注入攻击测试
- SQL注入
- XSS攻击
- CSRF攻击
- 命令注入

#### 重复攻击测试
- 重复提交订单
- 重复支付回调
- 重复申请退款
- 暴力破解订单ID

#### 信息泄露测试
- 敏感信息脱敏（银行卡号、手机号）
- 支付信息泄露
- 日志信息泄露

### 7. 测试点数量控制规则

- 每个功能点至少4个测试点（正向1、负向1、边界1、异常1）
- 关键功能点增加1-2个性能测试点
- 高风险功能点增加1-2个安全测试点
| 总测试点数量控制在功能点数量的4-8倍之间

### 测试维度平衡要求

> **重要：** 生成测试点时必须控制各维度的数量比例，避免单一维度占比过高。

**各维度数量占比建议（基础4维）：**
| 测试维度 | 建议占比 | 说 明 |
|---------|---------|------|
| 正向测试 | 25%-30% | 覆盖正常业务流程和核心功能 |
| 负向测试 | 25%-30% | 覆盖非法输入、异常操作、错误数据 |
| 边界测试 | 20%-25% | 覆盖数值边界、长度边界、数量边界 |
| 异常测试 | 20%-25% | 覆盖网络异常、服务异常、超时等场景 |

**性能与安全维度的触发条件（含性能/安全时）：**
| 维度 | 触发条件 | 建议占比 |
|------|---------|---------|
| 性能测试 | 需求文档中含有：并发、高并发、大量、响应时间、延迟、| 5%-10% |
| | TPS、QPS、吞吐量、效率、性能等关键词 | |
| 安全测试 | 需求文档中含有：权限、越权、安全、加密、敏感、| 5%-10% |
| | 认证、鉴权、Token、注入、XSS、非法访问等关键词 | |
| | 或涉及资金、用户隐私、核心数据的功能 | |

**检查清单：**
- [ ] 正向测试不超过总量 30%（如果超过，需合并或删除冗余的正向测试点）
- [ ] 负向/边界/异常各有覆盖（每个模块至少各 1 个）
- [ ] 安全测试已在高风险功能点中生成
- [ ] 性能测试已在效率敏感功能点中生成

## 注意事项

1. 测试点要具体，避免"验证功能"这种模糊表述
2. 每个测试点要有测试数据和预期结果
3. 测试维度要覆盖全面，避免遗漏
4. 对于高风险功能，要增加安全测试点
5. 对于性能敏感功能，要增加性能测试点
6. 测试点数量要合理，避免过度测试
7. 测试点要可执行，避免无法实现的情况
8. 性能测试要明确关键指标
9. 安全测试要覆盖高风险场景
10. 优先级测试点要特别标注

## 常见问题

**Q: 测试点太多怎么办？**
A: 可以根据业务优先级和风险等级，筛选重点测试点，降低次要测试点的密度。

**Q: 测试点太少怎么办？**
A: 检查是否遗漏了测试维度，特别是负向测试、边界测试和异常测试。

**Q: 如何确定测试点的优先级？**
A: 根据功能的重要性、风险等级、业务价值来分配测试点优先级。

**Q: 性能测试的标准是什么？**
A: 根据业务需求设定，一般关键操作<500ms，一般操作<1s。

**Q: 安全测试覆盖哪些场景？**
A: 越权访问、数据篡改、注入攻击、重复攻击、信息泄露等。

**Q: 测试点如何执行？**
A: 测试点是测试设计的输入，需要进一步转化为具体的测试用例才能执行。
