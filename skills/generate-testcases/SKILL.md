# 生成测试用例 Skill

基于测试点清单，生成结构化的测试用例文件，支持 Excel 和 XMind 两种格式输出。

## 触发条件

当用户说以下内容时，使用本 Skill：
- "生成测试用例"
- "生成 Excel 用例"
- "生成 XMind 用例"
- "把测试点转成用例"
- 带有 "generate-testcases"、"测试用例生成" 等关键词

## 执行步骤

### 1. 读取测试点文件
使用 `read_file` 工具读取 `testpoints.md` 文件。

### 2. 确认输出格式

支持两种格式：

**Excel 格式（默认）：**
```
用户：生成测试用例，Excel 格式
用户：生成 Excel 用例
```

**XMind 格式：**
```
用户：生成测试用例，XMind 格式
用户：生成 XMind 用例
```

### 3. 确认测试维度过滤（可选）

可以按测试维度过滤：
```
# 只生成正向测试用例
用户：生成用例，只包含正向测试

# 只生成基础测试用例（正向、负向、边界、异常）
用户：生成用例，只包含基础测试

# 包含所有测试维度
用户：生成用例，包含性能和安全测试
```

### 4. 生成测试用例

#### 4.1 Excel 格式

使用 Python 脚本生成 `.xlsx` 文件。

**Excel 列结构：**

| 列 | 字段名 | 说明 |
|----|--------|------|
| A | 用例编号 | TC-001 格式，自动编号 |
| B | 所属模块 | 功能模块名称 |
| C | 功能点 | 功能点名称 |
| D | 测试维度 | 正向/负向/边界/异常/性能/安全 |
| E | 用例标题 | 简洁的用例描述 |
| F | 优先级 | P0/P1/P2 |
| G | 前置条件 | 执行用例前的必要条件 |
| H | 测试步骤 | 编号步骤，每步一行 |
| I | 测试数据 | 具体的测试数据 |
| J | 预期结果 | 具体的预期结果 |
| K | 备注 | 额外说明（可选） |
| L | 执行结果 | 留空，人工执行后填写（通过/失败/阻塞/跳过） |

**Excel 格式化要求：**
- 表头加粗，背景色为深蓝色，白色字体
- 列宽自适应
- 优先级为 P0 的行用红色标注
- 优先级为 P1 的行用橙色标注
- 冻结首行

#### 4.2 XMind 格式

使用 Python 脚本生成 `.xmind` 文件。

**XMind 结构：**

```
[项目名称] 测试用例
├── [模块1名称]
│   ├── [功能点1.1]
│   │   ├── 正向测试
│   │   │   ├── TC-001: [用例标题] [P0]
│   │   │   │   ├── 前置条件: ...
│   │   │   │   ├── 测试步骤: ...
│   │   │   │   ├── 测试数据: ...
│   │   │   │   └── 预期结果: ...
│   │   │   └── TC-002: [用例标题] [P1]
│   │   ├── 负向测试
│   │   │   └── TC-003: ...
│   │   └── 边界测试
│   │       └── TC-004: ...
│   └── [功能点1.2]
│       └── ...
├── [模块2名称]
│   └── ...
└── 统计信息
    ├── 总用例数: N
    ├── P0: N
    ├── P1: N
    └── P2: N
```

### 5. 用例生成规则

#### 5.1 用例编号规则
- 格式：`TC-XXX`（三位数字，从001开始）
- 按模块和功能点顺序编号
- 编号连续不跳号

#### 5.2 优先级分配规则

脚本 `__assign_priority()` 基于 **核心模块 + 维度 + 标题关键词** 三级决策。详见 `references/steps-template-strategy.md`。

**P0（高优先级）— 至少满足一条：**
- 核心模块/功能点（用户管理、认证、支付、全流程等）的正向测试
- 核心模块的异常/关键负向/关键边界测试
- 涉及资金/安全的测试（越权、篡改、注入、泄露等）
- 校验/验证类、集成/串联/全流程类正向测试

**P1（中优先级）：**
- 一般功能的正向测试
- 大部分负向测试、异常测试
- 关键功能的边界测试、核心模块性能测试

**P2（低优先级）：**
- 非核心功能的性能测试
- UI 细节测试

> ⚠️ 核心模块定义（`CORE_MODULES` / `CORE_FEATURES`）硬编码在脚本中。若被测系统核心模块名不在列表中，需手动补充，否则该模块正向测试不会被提升为 P0。

#### 5.3 用例标题规则
- 格式：`[操作对象] + [操作动作] + [预期/场景]`
- 示例：
  - ✅ `用户使用正确账号密码登录成功`
  - ✅ `库存不足时下单提示库存不足`
  - ❌ `验证下单功能`
  - ❌ `下单测试`

#### 5.4 测试步骤规则
- 编号格式：`1. 2. 3.`（每个步骤一行）
- 每个步骤必须包含具体操作
- 每个步骤必须包含操作对象（按钮/链接/输入框）
- 示例：
  - ✅ `1. 在用户名输入框输入"testuser"`
  - ✅ `2. 在密码输入框输入"123456"`
  - ✅ `3. 点击"登录"按钮`
  - ❌ `1. 登录系统`

#### 5.5 预期结果规则
- 必须包含明确的判断标准
- 必须包含具体的提示信息（如有）
- 必须包含具体的状态变更（如有）
- 示例：
  - ✅ `登录成功，页面跳转到首页，右上角显示用户名"testuser"`
  - ✅ `提示"密码错误，请重新输入"，停留在登录页面`
  - ❌ `登录成功`
  - ❌ `提示错误`

## Python 脚本说明

### ⚠️ 运行环境（重要）

脚本依赖 `openpyxl`，该库**仅安装在 Hermes venv 中**，系统 `python3` 没有它。必须用 Hermes venv Python 运行：

```bash
HERMES_PYTHON="$HOME/.hermes/hermes-agent/venv/bin/python"
$HERMES_PYTHON scripts/generate_excel.py testpoints.md -o testcases.xlsx
$HERMES_PYTHON scripts/generate_xmind.py testpoints.md -o testcases.xmind
```

如果系统 `python3` 报 `ModuleNotFoundError: No module named 'openpyxl'`，不是安装问题——切换到 venv Python 即可。

### Excel 生成脚本
- 脚本位置：`scripts/generate_excel.py`
- 依赖：`openpyxl`（已在 Hermes venv 中安装）
- 功能：
  - 读取测试点 Markdown 文件
  - 解析模块、功能点、测试维度、测试点
  - 生成结构化的 Excel 文件（12列，含执行结果列）
  - 自动格式化：表头深蓝白字、P0红/P1橙/P2绿行标、冻结首行、自动筛选
  - 支持维度过滤：`-d positive,negative,basic,all`（中英文均可）

### XMind 生成脚本
- 脚本位置：`scripts/generate_xmind.py`
- 依赖：**[XMind](https://pypi.org/project/XMind/)（v1.2.0）** — 已安装在 Hermes venv 中
- 安装：`pip install XMind`
- 技术原理：使用 xmind 库生成标准 XMind 2.0 格式的 `.xmind` 文件（zip 包内含 content.xml + styles.xml + comments.xml）
- 功能：
  - 读取测试点 Markdown 文件
  - 生成树状脑图：项目 → 模块 → 功能点 → 测试维度 → 用例
  - 每个用例节点包含前置条件、测试数据、预期结果子节点
  - 末尾自动生成统计节点（P0/P1/P2 分布）

## ⚠️ 已知问题与陷阱

### 1. ~~步骤模板过于通用~~（v1.2.0 已修复）

> **v1.2.0 修复方案：** `__generate_steps()` 改用 `ACTION_TEMPLATES` 统一动作映射表（16 类操作模板，每类含正向+负向双步骤集），通过 `__match_action()` 统一匹配。未匹配时利用 `模块名 + 功能点 + 测试数据` 生成半具体化兜底步骤，不再回退到空泛的"执行目标操作"。实测通用模板回退率从 19% 降至 1%。详见 `references/steps-template-strategy.md`。

**仍有改进空间：** 少数用例（如"知识库检索响应时间"等标题不含动作动词的性能测试）仍可能匹配到较泛化的步骤。**在生成后建议人工抽查步骤列，对极少量兜底用例补充具体操作。**

### 2. ~~P0 优先级分配异常~~（v1.2.0 已修复）

> **v1.2.0 修复方案：** `__assign_priority()` 新增 `CORE_MODULES` / `CORE_FEATURES` / `CORE_ACTION_KW` 三组核心定义，实现多级决策：核心模块的正向/异常/关键负向/关键边界测试 → P0。实测 P0 占比从 15% 提升至 38%，P1 从 80% 降至 58%，P0 维度覆盖从 2 个扩展到 5 个。

**配置说明：** `CORE_MODULES` 和 `CORE_FEATURES` 是硬编码在脚本中的。如果被测系统的核心模块名称不在列表中（如医疗系统的"病历管理"），需要在脚本中补充对应关键词，否则这些模块的正向测试不会被提升为 P0。

### 3. 安全测试维度难以触发

安全测试维度仅在标题包含"越权""注入""XSS""SQL""篡改"等关键词时才生成对应步骤和优先级。如果测试点标题不包含这些词，安全维度不会生成有效用例。

### 4. 维度分布不平衡

正向测试占比通常过高（70-80%），负向/边界/异常/性能/安全严重偏少。脚本没有维度数量上限控制。这在测试报告中会被 `test-case-review` 的评审报告准确指出。

### 5. XMind 脚本不支持 `-d` 维度过滤参数

Excel 脚本支持 `-d positive,negative,basic,all` 维度过滤，但 XMind 脚本 `generate_xmind.py` 的 argparse 未定义 `-d` 参数，传参会报 `error: unrecognized arguments: -d all`。

**规避：** 如需维度过滤后生成 XMind，先用 Excel 脚本过滤生成，再手动转换；或修改 `generate_xmind.py` 添加 argparse `-d` 定义与 Excel 脚本对齐。

**修复方向：** 两个脚本 argparse 参数定义对齐（`-d`、`-o`、`-n` 等）。

## 注意事项

1. 测试用例基于测试点生成，测试点的质量直接影响用例质量
2. 生成的用例建议使用 test-case-review Skill 进行评审
3. Excel 文件需要使用 Excel 或 WPS 打开
4. XMind 文件需要使用 XMind 软件打开
5. 生成后可根据实际情况手动调整用例内容
6. 测试数据为示例数据，实际执行时需要根据环境调整
